# MedicalAgent 项目面试实现详解

## 1. 面试开场时可以怎么介绍这个项目

如果面试官先让你用 1 分钟介绍项目，可以这样讲：

> 这是一个基于 `Spring Boot + Spring AI + Spring AI Alibaba Agent Framework` 实现的医疗问答后端项目。  
> 它不是单纯调用大模型聊天，而是把医疗问答拆成了几个核心能力：会话管理、用户长期画像、RAG 知识检索、结构化诊断报告生成、就近医院与路线规划、PDF 导出。  
> 在实现上，我重点解决了三个问题：第一，如何把大模型问答做成稳定可控的后端服务；第二，如何把 RAG 从“能跑”优化到“更适合医疗检索”；第三，如何把聊天结果沉淀成可复用的结构化报告和规划结果。  
> 整个系统的统一门面是 `MedicalApp`，运行时底层是 `ReactAgent`，RAG 通过自定义 Hook 和 Interceptor 嵌入 Agent 生命周期，报告则通过快照机制保证查询、预览、PDF 导出一致。

如果面试官让你用 3 分钟介绍项目，可以在上面基础上补三点：

- 用户对外只感知 `sessionId`，后端内部维护 `threadId + userId`，把会话、画像和模型上下文解耦。
- RAG 不是直接用默认 Advisor，而是做了知识入库、查询增强、ES 混合检索、上下文注入、离线评估整套链路。
- 报告系统不是生成一段文本，而是先生成结构化模型，再做医院规划、快照冻结、PDF 导出，工程可扩展性更强。

## 2. 项目背景与业务目标

这个项目面对的不是普通聊天机器人，而是“医疗问答辅助”场景，所以目标比通用聊天更复杂：

1. 回答要尽量结构化，不能只有一段自然语言。
2. 要能识别高风险、信息不足、需要补充信息等医疗场景特征。
3. 要尽量利用知识库，提高回答的稳定性和可解释性。
4. 对话结束后还要能产出结构化报告，甚至延伸到医院规划和 PDF 导出。
5. 前端不能直接感知模型细节、线程细节或 RAG 实现细节。

所以项目最终落成的是一个“面向前端 API 的医疗 Agent 后端”，而不是“单次 prompt 调用脚本”。

## 4. 技术选型与技术对比

这一部分是面试里最容易被追问的，你不能只说“我用了什么”，还要说明“为什么不用别的”。

### 4.1 为什么选 `Spring Boot`

#### 选择原因

- 项目本质是标准后端服务，天然适合 Spring Boot。
- 需要同时承载 Web 接口、配置绑定、异步线程池、测试体系、依赖注入。
- Spring 生态和 Spring AI 结合非常自然，配置与 Bean 生命周期可控。

#### 和其他方案对比

| 方案 | 优点 | 缺点 | 本项目结论 |
| --- | --- | --- | --- |
| `Spring Boot` | 工程化成熟，和 Spring AI 生态一致 | 相对比轻框架更重 | 适合 |
| `Vert.x / Netty 手写服务` | 性能高、灵活 | 工程成本高，AI 业务层装配复杂 | 不适合当前阶段 |
| `Node.js + AI SDK` | 快速试错、前后端统一语言 | Java 项目迁移成本高，测试和工程治理需重建 | 不选 |

### 4.2 为什么选 `Spring AI + Spring AI Alibaba Agent Framework`

#### 选择原因

项目需要的不只是调用模型，而是：

- 统一 `ChatModel / EmbeddingModel / VectorStore` 抽象
- 支持 RAG ETL 和检索
- 支持 Agent 生命周期 Hook
- 支持模型调用拦截器
- 支持 MCP 工具接入

`Spring AI` 负责 AI 基础组件，`Spring AI Alibaba Agent Framework` 负责 Agent 运行时，两者配合刚好覆盖项目需求。

#### 和替代方案对比

| 方案 | 优点 | 缺点 | 本项目结论 |
| --- | --- | --- | --- |
| `Spring AI + Agent Framework` | 和 Spring 生态一致，适合 Java 后端工程化，支持 RAG、MCP、Hook、Interceptor | 部分能力需要自己做二次封装 | 最适合 |
| `只用 DashScope SDK 手写` | 依赖少，调用直接 | 要自己处理 Prompt、结构化输出、向量库抽象、工具编排 | 不利于长期扩展 |
| `LangChain4j` | Agent 和 RAG 生态成熟 | 与当前 Spring AI / Alibaba 路线不一致，切换成本高 | 可选但非本项目路线 |
| `只用 Spring AI ChatClient` | 轻量、简单 | 难以表达复杂 Agent 生命周期和自定义 Hook 链路 | 不足以覆盖项目需求 |

### 4.3 为什么选 `DashScope / Qwen`

#### 选择原因

- 项目面向中文医疗问答，中文表现和阿里生态接入都比较自然。
- 既要聊天模型，也要向量模型，DashScope 都能提供。
- 已经引入了 `spring-ai-alibaba-starter-dashscope`，接入链路统一。

#### 对比

| 方案 | 优点 | 缺点 | 本项目结论 |
| --- | --- | --- | --- |
| `DashScope + Qwen` | 中文能力较强，Spring AI Alibaba 支持完善 | 和国际模型生态不完全通用 | 当前最佳 |
| `OpenAI` | 通用性强、生态成熟 | 当前项目已落在阿里生态，切换成本高 | 非当前主选 |
| `本地模型` | 数据可控、成本可控 | 需要自建推理和向量能力，工程复杂 | 未来可评估 |

### 4.4 为什么选 `ReactAgent`

#### 选择原因

项目不是“单次问答”，而是需要：

- 会话线程
- 用户记忆
- 前置检索
- 后置画像提取
- 模型调用审计
- 工具调用

`ReactAgent` 适合把这些增强逻辑挂到统一生命周期里，而不是在 Controller 或 Service 层堆很多 if/else。

#### 对比

| 方案 | 优点 | 缺点 | 本项目结论 |
| --- | --- | --- | --- |
| `ReactAgent` | 生命周期清晰，适合插入 Hook/Interceptor/Tool | 需要理解 Agent 运行时机制 | 最适合 |
| `普通 Service 里串流程` | 直观 | 逻辑容易膨胀，扩展新能力会越来越乱 | 不利于长期维护 |
| `只用 Prompt + ChatModel` | 简单 | 无法优雅承载记忆、工具、RAG 注入链路 | 不够用 |

### 4.5 为什么默认向量库选 `Elasticsearch`

#### 选择原因

项目不是只需要“存向量”，还需要：

- 标准向量检索
- 词法检索
- 混合检索
- 工程上容易部署和观测

Elasticsearch 同时具备向量检索和 BM25 检索能力，很适合做“向量 + 词法”的混合召回。

#### 对比

| 方案 | 优点 | 缺点 | 本项目结论 |
| --- | --- | --- | --- |
| `Elasticsearch` | 同时支持向量检索和词法检索，适合混合召回 | 部署比内存方案重 | 默认主选 |
| `SimpleVectorStore` | 本地轻量、零外部依赖 | 只适合本地开发，不适合复杂检索 | 作为本地 fallback |
| `Milvus` | 向量能力强 | 词法检索要额外补方案 | 当前不是最优 |
| `pgvector` | 数据库统一管理方便 | 检索和混合召回能力通常需要额外设计 | 可做后续替代 |

### 4.6 为什么保留 `SimpleVectorStore`

这是一个很适合面试表达“工程弹性”的点。

虽然正式环境默认使用 Elasticsearch，但本地开发和演示环境并不一定方便起 ES，所以项目保留了：

- `medical.rag.vector-store.type=simple`
- ES 启动失败时可按配置回退到 SimpleVectorStore

这样做的价值是：

- 本地开发门槛更低
- Demo 场景可快速跑通
- 不会把所有环境都强耦合到 ES

### 4.7 为什么 RAG 不直接用标准 Advisor，而是自定义 Hook + Interceptor

这是本项目的一个核心设计点。

#### 如果只用标准 Advisor

优点：

- 接入快
- 代码少

缺点：

- 不容易和 Agent 生命周期深度融合
- 检索触发策略、上下文缓存、来源回传、日志审计都不够灵活
- 难以做项目定制化的查询增强和 ES 混合检索

#### 当前方案

项目选择：

- 在 `MedicalRagAgentHook` 决定是否检索
- 在 `MedicalRagContextInterceptor` 注入上下文
- 在 `MedicalRagContextHolder` 里缓存当前线程的 RAG 结果

这样做的收益是：

- RAG 可以成为 Agent 生命周期中的“正式一环”
- 能精细控制“什么时候查、查什么、怎么改写、怎么注入、怎么返回给前端”

### 4.8 为什么医院规划用 `MCP`

#### 选择原因

医院规划不是只要“调一个地图接口”，而是需要：

- 查询周边医院
- 医院详情补充
- 逆地理编码
- 步行/驾车/公交路线规划
- 后续还可能替换工具提供方

MCP 的好处是把这些工具调用能力抽象成统一工具协议，便于 Agent 侧复用。

#### 对比

| 方案 | 优点 | 缺点 | 本项目结论 |
| --- | --- | --- | --- |
| `MCP` | 工具抽象统一，适合 Agent 化调用 | 需要额外理解协议和工具配置 | 当前主选 |
| `直接 HTTP 调地图 SDK` | 简单直接 | 工具扩展性差，不利于 Agent 工具复用 | 可做底层替代，但不是当前主路线 |

### 4.9 为什么 PDF 用 `OpenPDF`

#### 选择原因

- 项目导出的是结构化文本型报告，不需要复杂网页渲染。
- 直接在后端渲染 PDF，调用链短，部署简单。
- 支持嵌入中文字体，满足当前导出需求。

#### 对比

| 方案 | 优点 | 缺点 | 本项目结论 |
| --- | --- | --- | --- |
| `OpenPDF` | 后端纯 Java 生成，简单直接 | 页面表现力一般 | 当前最适合 |
| `HTML + 浏览器打印 PDF` | 样式更自由 | 需要额外浏览器运行环境 | 当前没必要 |
| `iText` | 功能强 | 商业授权和成本考虑 | 当前不选 |

## 5. 整体系统架构

### 5.1 分层结构

```text
Controller 层
  -> Session 会话层
  -> MedicalApp 统一门面
  -> Service 业务编排层
  -> Agent Runtime 层
  -> Hook / Interceptor 增强层
  -> RAG 子系统
  -> Repository / Memory 存储层
  -> 外部能力：DashScope / Elasticsearch / MCP / PDF
```

### 5.2 模块职责划分

- `controller / controller.v1`：暴露 HTTP 接口
- `session`：维护 `sessionId -> threadId + userId + location`
- `app.MedicalApp`：统一业务门面
- `app.service.chat`：聊天主链路
- `app.service.profile`：画像服务
- `app.service.report`：报告、规划、PDF、快照
- `app.service.runtime`：Agent 运行时装配
- `app.rag.*`：知识库入库、检索、注入、评估
- `hook / interceptor`：业务级模型增强
- `repository*`：内存态存储

## 6. 核心数据模型设计

### 6.1 为什么要分 `sessionId / threadId / userId`

这是面试非常值得讲的一个设计点。

#### 角色定义

- `userId`：用户标识，绑定长期画像
- `sessionId`：前端会话标识，对前端暴露
- `threadId`：Agent 对话线程标识，对模型链路使用

#### 这样设计的好处

1. 前端不需要感知内部 Agent 线程机制。
2. 同一个用户可以开启多个会话，画像还能复用。
3. 会话态信息，比如经纬度、最后活跃时间，可以挂在 `session` 上，不污染模型线程结构。

#### 具体实现

- `ConsultationSessionService.createSession(userId, threadId)` 创建前端会话
- `ConsultationSessionRepository` 用于存取会话
- `ChatController` 收到 `sessionId` 后，通过 `ConsultationSessionService.getRequiredSession` 找到对应 `threadId + userId`

## 7. 聊天主链路实现细节

### 7.1 请求入口

聊天接口由 `ChatController` 提供，支持两种返回形式：

- 同步 JSON
- SSE 伪流式

#### 为什么同时支持两种

- JSON 方便普通接口调试与稳定集成
- SSE 方便前端做聊天打字机体验

### 7.2 为什么当前 SSE 是“伪流式”

项目现在的 SSE 不是模型 token 级流式，而是：

1. 先同步拿到完整模型回复
2. 再按标点和长度切分成多个 chunk 推给前端

#### 这样做的原因

- 降低前端对真实流式协议的处理复杂度
- 保持当前业务链路简单稳定
- 和结构化封装逻辑更容易兼容

#### 局限

- 体验上像流式，但本质不是模型边生成边返回
- 如果追求更低首字延迟，后面可以升级成真实流式

### 7.3 聊天业务编排

聊天核心在 `DefaultMedicalChatService`。

它主要做这些事：

1. 规范化 `threadId` 和 `userId`
2. 判断用户当前输入是否是在“请求生成报告”
3. 如果不是报告请求，就调用 `MedicalAgentRuntime.doChatMessage`
4. 对模型返回做 `MedicalReplyFormatter.normalize`
5. 把用户消息和助手消息追加进线程历史
6. 读取当前线程的 RAG 上下文
7. 基于结构化回复和会话历史计算报告触发等级
8. 封装成统一的 `MedicalChatResult`

### 7.4 为什么单独做 `MedicalReplyFormatter`

大模型的自然语言输出存在不稳定性，所以项目没有直接把原始文本返回，而是加了一层规范化：

- 清理 Markdown 标题、代码块、粗体等格式噪声
- 尝试按“风险等级 / 核心判断 / 主要依据 / 建议下一步 / 何时就医 / 需要补充 / 免责声明”这些标签解析
- 解析失败时做降级兜底

这层的价值是：

- 提升前端展示的一致性
- 让报告触发策略能依赖结构化字段
- 降低 Prompt 偶发漂移带来的影响

## 8. Agent Runtime 设计

### 8.1 Runtime 的核心职责

`DefaultMedicalAgentRuntime` 负责把底层 Agent 真正组装起来。

它在初始化 `ReactAgent` 时做了几件关键事：

- 指定聊天模型：`medicalAiModelProvider.getChatModel()`
- 指定系统 Prompt：`MedicalPrompts.MEDICAL_AGENT_INSTRUCTION`
- 挂载业务 Hook：`MedicalRagAgentHook`、`UserProfileMemoryHook`
- 按配置可选挂载 `ReReadingMessagesHook`
- 挂载 Interceptor：`MedicalSystemPromptInterceptor`、`MedicalRagContextInterceptor`、`MyLogModelInterceptor`
- 挂载 `MemorySaver`

### 8.2 Hook 和 Interceptor 的职责边界

这是面试里很适合展示你对 Agent 生命周期理解的点。

#### Hook 更适合做什么

- 在 Agent 开始前决定是否检索
- 在模型后提取用户画像
- 改写消息列表

#### Interceptor 更适合做什么

- 修改系统提示词
- 注入上下文
- 记录请求/响应日志

#### 本项目的具体分工

- `MedicalRagAgentHook`：决定是否执行 RAG
- `UserProfileMemoryHook`：模型后回收用户长期信息
- `ReReadingMessagesHook`：可选重写最后一条用户消息
- `MedicalSystemPromptInterceptor`：把用户画像上下文拼到系统消息
- `MedicalRagContextInterceptor`：把检索结果拼到系统消息
- `MyLogModelInterceptor`：打印模型请求和响应摘要

### 8.3 为什么 `ReReadingMessagesHook` 是可配置开关

Re-reading 本质上是一种提示增强技巧，可以在某些场景提升模型对当前问题的聚焦度，但它不一定在所有场景都稳定，所以项目通过：

- JVM 属性 `medical.agent.rereading.enabled`
- 环境变量 `MEDICAL_AGENT_REREADING_ENABLED`

把它做成了运行期开关，而不是硬编码。

这种做法的工程价值是：

- 可灰度开启
- 便于 A/B 验证
- 出现副作用可快速关闭

## 9. 用户长期画像实现细节

### 9.1 为什么要做用户画像

医疗问答场景里，很多信息是跨轮稳定的，比如：

- 姓名
- 年龄
- 性别
- 过敏史

如果每次都让用户重复输入，体验很差；如果完全交给模型“自己记忆”，又不稳定。

所以项目引入了“后端显式画像存储”。

### 9.2 画像怎么存

当前采用 `UserProfileRepository` 抽象 + `MemoryStoreUserProfileRepository` 实现。

原因是：

- 当前项目仍偏原型和验证阶段
- 先把接口抽象出来，后续可切 Redis/MySQL/文档数据库
- 先保证业务闭环，再演进持久化方案

### 9.3 画像怎么提取

采用“双通道提取”：

#### 第一层：规则提取

`DefaultUserProfileService` 用正则提取：

- 姓名
- 年龄
- 性别
- 过敏史

优点：

- 快
- 可控
- 无模型成本

#### 第二层：结构化 LLM 提取

当规则提取不够完整时，调用 `DashScopeUserProfileFactExtractor`：

- 使用结构化输出
- 只允许返回固定字段
- 用 `confidence` 控制结果可信度
- 置信度低于阈值则丢弃

#### 为什么这样设计

如果全靠规则，覆盖率不够；如果全靠模型，成本高且不稳定。  
所以项目采用“规则优先，模型补充”的折中方案。

### 9.4 画像怎么注入模型

在 `MedicalSystemPromptInterceptor` 中：

1. 从请求上下文读取 `user_id`
2. 调用 `UserProfileService.buildProfileContext`
3. 把画像信息拼接到系统消息

这样做的好处是：

- 模型使用的是“显式注入的画像上下文”
- 不需要依赖黑盒对话记忆
- 用户换会话时也能继承长期资料

## 10. RAG 实现细节

这是面试里最值得展开讲的一块。

### 10.1 设计目标

项目中的 RAG 不是为了“有个知识库”而已，而是为了解决三个问题：

1. 医疗问答要尽量减少幻觉
2. 医疗问题很多是 follow-up，需要更智能地构造查询
3. 单纯向量检索在医疗文本里不一定稳定，所以要支持混合召回

### 10.2 官方基线

相关官方文档：

- RAG 总体能力：<https://java2ai.com/integration/rag/retrieval-augmented-generation/>
- ETL Pipeline：<https://java2ai.com/integration/rag/etl-pipeline/>
- Elasticsearch 向量库：<https://java2ai.com/integration/rag/vectordbs/elasticsearch>

官方基线思路主要是：

- 用 `DocumentReader -> Transformer -> VectorStore` 完成 ETL
- 用标准 `VectorStore` 完成检索
- 在生成时把检索结果注入回答链路

### 10.3 当前项目为什么做了自定义 RAG

项目保留的自定义点有：

1. `simple | elasticsearch` 双向量库切换  
   原因：支持轻量本地运行和正式 ES 环境切换。

2. `StableDocumentIdTransformer + KnowledgeManifestRepository`  
   原因：重建知识库时要先删除旧文档，避免索引污染。

3. `ElasticsearchBackedVectorStore`  
   原因：需要手动控制 ES 启动失败后的 fallback。

4. `ElasticsearchHybridSearchClient`  
   原因：项目要做向量 + BM25 + RRF 融合。

5. `MedicalRagAgentHook + MedicalRagContextInterceptor`  
   原因：项目要把 RAG 深度嵌到 Agent 生命周期里。

### 10.4 RAG ETL 入库实现

入库主流程在 `DefaultKnowledgeIngestionService`。

#### ETL 流程

```text
读取文档
  -> 元数据规范化
  -> 文本切块
  -> 生成稳定 ID
  -> 可选摘要富化
  -> 可选关键词富化
  -> 分批写入向量库
  -> 保存 manifest
```

#### 关键实现点 1：文档读取

`MedicalKnowledgeDocumentLoader` 负责读取 `resources/knowledge-base/medical` 下的 `.md/.txt` 文件：

- Markdown 用 `MarkdownDocumentReader`
- TXT 用 `TextReader`
- 对 Markdown front matter 做轻量预处理

#### 关键实现点 2：元数据规范化

`MedicalKnowledgeMetadataNormalizerTransformer` 统一输出：

- `sourceId`
- `title`
- `section`
- `uri`

这样做的目的是保证后续检索、来源展示、去重逻辑都依赖统一字段。

#### 关键实现点 3：稳定文档 ID

`StableDocumentIdTransformer` 使用：

- `sourceId`
- `section`
- `chunk_index`

做 SHA-256 截断生成稳定 ID。

为什么要这样做：

- 同一批文档重新入库时 ID 必须稳定
- 才能通过 manifest 删除上一批旧文档
- 避免索引越重建越脏

#### 关键实现点 4：分批写入

`DefaultKnowledgeIngestionService` 把写入批大小控制在 `25`，原因是：

- DashScope embedding 接口每次最多接收 25 条文本
- 所以 ETL 写入必须主动分批

这类细节很适合面试里讲，能体现你不是只会“调用框架”，而是真的处理过外部能力约束。

### 10.5 向量库装配实现

`MedicalRagConfiguration` 负责创建 `VectorStore`。

#### 支持两种模式

1. `simple`
2. `elasticsearch`

#### 为什么自定义 ES 装配

Spring AI 官方是支持 `ElasticsearchVectorStore` 的，但项目做了额外包装：

- 手动执行 `afterPropertiesSet`
- 如果 ES 启动时连接失败，可按配置回退到 `SimpleVectorStore`
- 用 `ElasticsearchBackedVectorStore` 避免 Bean 生命周期重复触发初始化

这说明项目并不是简单“自动配置一把梭”，而是做了工程增强。

### 10.6 Embedding 模型的 query/document 区分

`MedicalQueryAwareEmbeddingModel` 是一个比较有亮点的实现。

很多项目直接用同一个 embedding 配置同时做入库和查询，但这里显式区分了：

- 文档向量：`document`
- 查询向量：`query`

原因：

- DashScope embedding 支持不同 `textType`
- 文档与查询的语义分布不同
- 区分有助于提升检索质量

### 10.7 检索触发策略

`DefaultRagTriggerPolicy` 不是“每条都检索”，而是做了规则过滤：

- 问候语不查
- 用户画像类输入不查
- 报告请求不查
- 太短的输入不查

原因：

- 减少无意义检索
- 降低模型与向量接口开销
- 防止无关知识污染系统提示词

### 10.8 检索查询构建

`DefaultMedicalQueryBuilder` 的策略是：

- 优先取最后一条用户问题
- 如果像 follow-up 追问，就拼接上一轮用户问题

比如用户问：

1. “我这两天一直头晕”
2. “那这种情况什么时候必须去医院？”

第二问单独拿出来做向量检索，语义太弱；拼上前文效果会更好。

### 10.9 检索查询增强

`DashScopeRetrievalQueryEnhancer` 会用 LLM 把用户问题改写成更适合知识库检索的查询。

它特别做了一个医疗场景优化：

- 否定信息只能作为辅助过滤条件，不能成为检索主题

举例：

- 原问题：“胸闷，没有发热，没有咳嗽”
- 强化后检索重点应该是“胸闷”，而不是“没有发热”

这说明项目对医疗检索的 query rewrite 有专门优化，不是简单“原文拿去查”。

### 10.10 检索执行

`DefaultMedicalKnowledgeRetriever` 支持两种检索模式：

#### 方案 A：纯向量检索

直接对 `VectorStore` 做 `similaritySearch`。

#### 方案 B：Elasticsearch 混合检索

`ElasticsearchHybridSearchClient` 会：

1. 先跑向量召回
2. 再跑词法召回（multi-match / BM25）
3. 最后用 `RRF` 融合

#### 为什么要混合检索

医疗文本里很多关键词非常关键，比如：

- 病名
- 检查名
- 症状实体
- 科室关键词

这些内容纯向量检索不一定稳，混入词法检索后鲁棒性会更好。

### 10.11 RAG 上下文注入

当检索得到 `RagContext` 后：

- `MedicalRagAgentHook` 把它写入 `RunnableConfig.metadata`
- `MedicalRagContextHolder` 按线程缓存
- `MedicalRagContextInterceptor` 在真正调用模型前，把 `[MEDICAL_RAG_CONTEXT]` 注入系统消息

这层设计的价值：

- 模型能显式看到知识依据
- 前端还能拿到 `sources`
- 聊天业务层还能知道本轮是否应用了 RAG

### 10.12 RAG 离线评估

`DefaultOfflineRagEvaluationService` 会读取 `medical-rag-eval.json`，批量执行检索并统计：

- 命中率
- MRR
- 关键词覆盖率

为什么这块重要：

- RAG 优化如果只靠主观感觉，很难持续迭代
- 有离线评估才能比较“调参前后到底好没好”

这是面试里很能加分的一点，因为很多项目“做了 RAG”，但没有评估闭环。

## 11. 结构化报告实现细节

### 11.1 为什么要做结构化报告

医疗问答最终如果只返回一段文本，会有几个问题：

- 前端难展示
- 无法导出 PDF
- 无法做医院规划
- 无法做后续数据复用

所以项目把报告设计成结构化对象 `MedicalDiagnosisReport`，而不是一段 Markdown。

### 11.2 报告生成入口

报告入口主要有两类：

1. 用户显式要求“生成报告”
2. 聊天过程中后端判断“已经达到生成报告条件”

### 11.3 报告触发策略

`DefaultMedicalReportTriggerPolicy` 会综合下面这些因素判断：

- 风险等级
- 是否包含高风险短语
- 是否已经有完整的 summary / basis / nextSteps / escalationSignals
- 是否还处于“信息不足，需要补充”状态
- 用户轮次数是否足够

#### 触发等级

- `NONE`
- `SUGGESTED`
- `RECOMMENDED`
- `URGENT`

#### 为什么要分等级

因为报告不是“只有生成/不生成”两种状态：

- 有些是建议生成
- 有些是明确推荐生成
- 有些高风险场景需要尽快生成辅助就医

### 11.4 报告生成方式

`DefaultMedicalReportService` 通过：

- 当前线程完整对话
- 用户长期资料
- 最近一轮助手回答

构造 Prompt，然后用 `BeanOutputConverter<MedicalDiagnosisReport>` 要求模型返回结构化 JSON。

#### 为什么选结构化输出

和生成自由文本相比，结构化输出的优势是：

- 字段稳定
- 便于前端展示
- 便于后续医院规划和 PDF 导出
- 出错时更容易做 fallback

### 11.5 报告生成失败怎么兜底

如果结构化报告生成失败，项目不会直接抛错，而是：

- 用当前助手回复做兜底
- 返回一个最小可用的 `MedicalDiagnosisReport`
- 带上 `uncertaintyReason`

这样做的好处是：

- 不让整个报告链路彻底失败
- 提升用户体验和系统韧性

## 12. 报告快照机制

这是项目里很有工程含量的一块。

### 12.1 为什么需要快照

如果每次：

- 查询报告
- 预览报告
- 下载 PDF

都重新调用模型生成一次报告，会有三个问题：

1. 成本高
2. 延迟高
3. 结果可能前后不一致

所以项目引入 `MedicalReportSnapshot`。

### 12.2 快照里存什么

- `sessionId`
- `threadId`
- `userId`
- 生成时间
- 对话指纹
- 画像指纹
- 位置指纹
- 结构化报告
- 医院规划结果

### 12.3 如何判断快照还能不能复用

`DefaultMedicalReportSnapshotService` 会计算三类 fingerprint：

- 对话 fingerprint
- 画像 fingerprint
- 位置 fingerprint

只要这三类都没变，就复用旧快照；否则重新生成。

### 12.4 并发问题怎么处理

项目还专门做了按 `sessionId` 的锁：

- `snapshotLocks`
- `ReentrantLock`
- stale lock cleanup

为什么做这个：

- 避免同一会话下并发请求同时触发重复生成
- 避免锁对象一直泄漏

这块很适合面试里展示“不是只有业务流程，还有并发与缓存一致性考虑”。

## 13. 医院规划实现细节

### 13.1 为什么把医院规划作为独立子系统

报告生成和医院规划虽然相关，但职责不同：

- 报告负责“医疗判断结构化”
- 规划负责“把判断转成就医行动建议”

所以项目把它拆成了独立能力：

- `MedicalPlanningIntentResolver`
- `MedicalHospitalPlanningService`
- `MedicalHospitalPlanningGateway`
- `MedicalHospitalPlanningAgent`

### 13.2 规划意图解析

`DefaultMedicalPlanningIntentResolver` 会结合：

- 用户最新输入
- 报告内容
- 风险等级
- 关键词

解析出一个 `MedicalPlanningIntent`，里面包括：

- 是否需要规划
- 是否用户显式要求
- 搜索 profile
- 医院关键词
- 医院类型
- 搜索半径
- topK
- 是否优先三甲

#### 为什么单独做一层 intent

因为后续不管是：

- local fallback
- 直接 MCP
- agentic MCP

都可以复用这层解析结果。

### 13.3 规划模式设计

`MedicalReportPlanningProperties` 支持四种模式：

- `off`
- `local`
- `mcp`
- `agentic`

#### 各模式含义

| 模式 | 含义 |
| --- | --- |
| `off` | 完全关闭医院规划 |
| `local` | 只用本地候选医院列表，按距离排序 |
| `mcp` | 用 MCP 工具做确定性医院搜索和路线规划 |
| `agentic` | 先让 Agent 自主调工具规划，失败再回退到确定性 MCP |

#### 为什么这样设计

因为地图规划能力的稳定性和部署复杂度并不是单一的：

- 本地开发可先用 `local`
- 正式联调可启用 `mcp`
- 想验证 Agent 工具调用能力时可以用 `agentic`

### 13.4 MCP 规划实现

`McpMedicalHospitalPlanningGateway` 是确定性 MCP 规划入口，核心步骤是：

1. 创建 `McpSyncClient`
2. 枚举可用工具
3. 找到周边搜索、逆地理编码、路线规划等工具名
4. 搜索周边医院
5. 必要时放宽关键词和半径
6. 查询详情和路线
7. 归一化结果

这层强调的是：

- 工具名不完全写死，有一定兼容性处理
- 结果会标准化成项目自己的 `MedicalHospitalPlanningSummary`

### 13.5 Agentic 规划实现

`DefaultMedicalHospitalPlanningAgent` 会：

- 基于 `ReactAgent`
- 只开放医院规划相关 MCP 工具
- 通过严格 JSON 输出返回规划结果

#### 为什么还要保留确定性 MCP fallback

因为 Agent 工具调用虽然更灵活，但结果稳定性不一定绝对可靠，所以项目做了二层保障：

1. 先尝试 agentic
2. 结果不可用就回退到 MCP
3. MCP 也失败再回退到 local fallback

这种多级降级是很典型的工程设计亮点。

### 13.6 本地降级方案

`DefaultMedicalHospitalPlanningService` 在以下场景会降级：

- 没有坐标
- 规划模式为 `local`
- MCP 不可用
- agentic 失败

降级方案会：

- 用配置里的候选医院列表
- 计算用户到医院的直线距离
- 按距离或三甲优先排序

这样即使地图工具不可用，系统仍能给用户一个最基本的可用结果。

## 14. PDF 导出实现细节

### 14.1 为什么 PDF 导出不是直接让大模型生成文件

因为 PDF 是工程输出，不是模型擅长的原生格式。

项目做法是：

1. 先生成结构化报告
2. 再交给后端 PDF 渲染器渲染

好处：

- 结果稳定
- 可控
- 易测试

### 14.2 渲染流程

`DefaultMedicalReportPdfExportService` 做导出服务层封装：

- 校验报告是否可导出
- 构造 `MedicalReportPdfPayload`
- 调用 `MedicalReportPdfRenderer`

`MedicalReportPdfRenderer` 使用 `OpenPDF`：

- 设定 A4 页面
- 嵌入中文字体
- 按 section 写入标题、风险等级、患者概述、主要依据、医院路线等内容

### 14.3 为什么要内置中文字体

如果不处理字体，中文 PDF 很容易乱码。  
项目把默认字体资源放在 `src/main/resources/fonts/HiraginoSansGB.ttc`，并支持通过配置覆盖字体路径。

这类细节面试里很容易体现你做过真实后端交付，而不是只停留在逻辑代码层。

### 14.4 为什么还做了 PDF Tool Factory

`MedicalReportPdfToolFactory` 把 PDF 导出能力封装成工具，是为了后续支持：

- Agent 直接调用工具导出
- 报告导出能力可复用

这是一种“先结构化，再工具化”的设计。

## 15. 异常处理、稳定性与可观测性

### 15.1 异常处理

`GlobalExceptionHandler` 统一处理：

- 参数校验异常
- session 不存在
- 附件不支持
- 模型配置异常
- GraphRunner 异常
- 上游网络异常
- PDF 导出异常

这样做的好处是：

- Controller 更干净
- 前端收到的错误格式更稳定
- 日志更便于排查

### 15.2 日志与可观测性

`MyLogModelInterceptor` 重点记录：

- threadId
- userId
- messageCount
- tools
- ragApplied
- ragQuery
- ragSources
- response 摘要

这让模型调用不再是黑盒，出了问题可以定位：

- 是不是走了 RAG
- 检索词是什么
- 检索到了哪些来源
- 模型返回了什么

项目测试覆盖面比较完整，这一块在面试里很加分。

### 16.1 测试层次

| 测试层次 | 代表测试 |
| --- | --- |
| 单元测试 | `MedicalReplyFormatterTest`、`DefaultMedicalReportTriggerPolicyTest` |
| 组件测试 | `DefaultUserProfileServiceTest`、`DefaultKnowledgeIngestionServiceTest` |
| 工作流测试 | `MedicalRagWorkflowTest`、`MedicalRagElasticsearchWorkflowTest` |

### 16.2 为什么要分普通测试和 Live 测试

所以项目通过命名和 Maven Profile 把它们区分开：

- 单元/普通测试默认执行
- `*IT` / `*LiveIT` 通过 Failsafe 或特定 profile 执行

这是一种比较成熟的 AI 工程测试策略。

## 17. 详细实现路线

如果面试官问你“这个项目是怎么一步步做出来的”，可以按这个路线讲。

### 第一步：先把标准 Web 后端骨架搭起来

目标：

- 跑通 Spring Boot
- 暴露用户初始化、聊天、报告查询等基础 API

先做的内容：

- `controller.v1`
- `ApiResponse`
- `GlobalExceptionHandler`
- `session` 管理

为什么先做这步：

- 先把对外 API 边界稳定下来
- 后续模型、RAG、报告都可以往里填

### 第二步：把聊天能力抽象成统一门面

目标：

- 不让 Controller 直接拼模型逻辑

先做的内容：

- `MedicalApp`
- `MedicalChatService`
- `MedicalAgentRuntime`

为什么这样做：

- 后续加画像、RAG、报告时，Controller 不会不断膨胀

### 第三步：接入大模型并建立 Agent Runtime

目标：

- 把“直接调模型”升级成“可插拔增强的 Agent 运行时”

做法：

- 接入 DashScope ChatModel
- 用 `ReactAgent` 统一承载对话
- 加入 `Hook / Interceptor`

### 第四步：补用户长期画像

目标：

- 提升跨轮和跨会话体验

做法：

- 存储画像
- 规则提取 + LLM 提取
- 在系统消息里注入画像上下文

### 第五步：补 RAG 基础链路

目标：

- 给医疗问答增加外部知识依据

做法：

- 建知识库目录
- 做 ETL 入库
- 接入向量库
- 实现检索和上下文注入

### 第六步：优化 RAG 检索质量

目标：

- 从“能检索”升级到“更适合医疗场景”

做法：

- 做查询触发策略
- 做 query rewrite
- 区分 query/document embedding
- 增加 ES hybrid search + RRF
- 做离线评估

### 第七步：把聊天结果沉淀成结构化报告

目标：

- 从聊天系统升级到“可沉淀结果的业务系统”

做法：

- 定义 `MedicalDiagnosisReport`
- 用结构化输出生成报告
- 做报告触发等级

## 18. 这个项目的工程亮点

面试时可以重点提下面这些亮点。

### 亮点 1：会话与模型线程解耦

前端只关心 `sessionId`，内部才使用 `threadId`，这是比较典型的“接口模型”和“内部执行模型”分离。

### 亮点 2：RAG 深度嵌入 Agent 生命周期

不是简单把检索放在 Service 层，而是通过 Hook + Interceptor 做成运行时能力。

### 亮点 3：RAG 检索质量优化比较完整

不是只有向量检索，还做了：

- query builder
- query enhancer
- hybrid search
- RRF
- 离线评估

### 亮点 4：结构化报告而不是纯文本报告

这样后续才能支持：

- 前端稳定展示
- 快照复用
- PDF 导出
- 医院规划

### 亮点 5：医院规划支持多级降级

`agentic -> mcp -> local fallback`

这类多级降级很能体现工程思维。

### 亮点 6：快照 + 指纹 + 锁机制

避免重复生成、保证内容一致、处理并发，这是很典型的后端系统设计亮点。

## 19. 面试时可能被追问的问题

### 19.1 为什么不用数据库存会话和画像

可回答：

> 当前项目阶段更偏原型验证和能力整合，优先解决业务闭环和链路稳定性，所以先用了内存实现；但仓储接口已经抽象出来了，后续替换 Redis/MySQL 的改动范围会比较可控。

### 19.2 为什么不直接让大模型生成最终报告文本

可回答：

> 因为最终还要支持报告查询、前端展示、PDF 导出、医院规划，所以必须先沉淀成结构化对象；自由文本不利于工程复用。

### 19.3 RAG 为什么要做 hybrid search

可回答：

> 医疗文本里有很多专业关键词和危险信号，纯向量检索在一些场景下不如“向量 + BM25”稳定，所以我把 Elasticsearch 的词法检索也用起来，再用 RRF 融合结果。

### 19.4 为什么报告要做快照

可回答：

> 如果每次预览、查询、下载都重新调模型，会导致成本高、延迟高、结果不一致，所以我做了报告快照，并用对话/画像/位置指纹来判断是否还能复用。

### 19.5 这个项目里你觉得最有技术含量的地方是什么

可回答：

> 我觉得主要有三块：  
> 第一是 RAG 从入库到混合检索到上下文注入的整条链路；  
> 第二是结构化报告 + 快照 + PDF 的工程化落地；  
> 第三是医院规划能力的多模式设计和降级策略。
