# MedicalAgent v1 后端接口说明

## 1. 文档目的

本文档描述当前后端已经实现的 `v1` 接口能力，供前端联调用。

这份说明以当前代码实现为准，不包含尚未落地的接口。

## 2. 基本信息

- 接口版本：`v1`
- 服务基路径：`/api`
- 对外接口前缀：`/api/v1`
- 返回格式：
  - 普通 JSON 接口：统一返回 `ApiResponse<T>`
  - SSE 接口：返回 `text/event-stream`，不使用 `ApiResponse` 包装
- 鉴权：当前无鉴权
- 持久化：当前 `session` 和用户画像为内存态，服务重启后会丢失

## 3. 当前支持的数据流转模式

### 模式 A：初始化资料 -> 同步聊天 -> 查询报告

1. 前端调用 `POST /api/v1/users/profile`
2. 后端创建或复用 `userId`，并始终创建一个新的 `sessionId`
3. 前端保存 `userId` 和 `sessionId`
4. 前端调用 `POST /api/v1/chat/completions`，`Accept: application/json`
5. 后端通过 `sessionId` 映射到内部 `threadId + userId`
6. 后端返回一次完整问诊结果
7. 前端按需调用 `GET /api/v1/reports/{sessionId}` 查询结构化报告

### 模式 B：初始化资料 -> SSE 聊天 -> 查询报告

1. 前端调用 `POST /api/v1/users/profile`
2. 前端调用 `POST /api/v1/chat/completions`，`Accept: text/event-stream`
3. 后端先发送 `meta` 事件
4. 后端生成完整回复后，再拆分为若干 `chunk` 事件推送
5. 后端最后发送 `done` 事件
6. 前端按需调用 `GET /api/v1/reports/{sessionId}` 查询报告

### 模式 C：同一用户开启新会话

1. 前端再次调用 `POST /api/v1/users/profile`，携带已有 `userId`
2. 后端更新用户画像
3. 后端重新创建一个新的 `sessionId`
4. 前端后续聊天应使用新的 `sessionId`

## 4. 当前数据流转约束

### 4.1 前端只使用 `userId + sessionId`

前端不需要也不会接触内部 `threadId`。

后端内部映射关系：

- `userId`：用户标识
- `sessionId`：前端会话标识
- `threadId`：后端内部问诊线程标识

当前关系是：

- 每次初始化资料都会生成一个新的 `sessionId`
- 每个 `sessionId` 绑定一个内部 `threadId`
- 同一个 `userId` 可以对应多个历史 `sessionId`

### 4.2 当前 SSE 不是模型真流式

当前 SSE 已经可用于前端流式渲染，但它的实现语义需要明确：

- 后端不是 token 级实时返回
- 后端会先同步拿到完整回复
- 再按“中文标点优先，超长 12 字切分”的规则拆成多个 `chunk`
- 因此前端能做打字机效果，但不应把它理解成模型原生流式输出

### 4.3 附件字段已预留，但当前不支持上传

聊天请求体里允许携带 `attachments` 字段，但当前只要非空就会返回错误：

- HTTP 400
- 错误信息：`暂不支持附件上传`

前端当前应传空数组或不传该字段。

## 5. 统一 JSON 响应结构

所有普通 JSON 接口统一返回：

```json
{
  "code": 200,
  "message": "OK",
  "data": {}
}
```

字段说明：

- `code`：与 HTTP 状态码一致
- `message`：响应说明
- `data`：业务数据，失败时通常为 `null`

## 6. 接口说明

### 6.1 初始化用户资料

- 方法：`POST`
- 路径：`/api/v1/users/profile`
- `Content-Type`：`application/json`
- 响应：`application/json`

#### 请求体

```json
{
  "userId": "usr_existing_optional",
  "name": "张三",
  "age": 28,
  "gender": "MALE",
  "avatarId": "memoji_01"
}
```

#### 字段说明

- `userId`：可选，不传则后端创建
- `name`：必填，不能为空
- `age`：必填，范围 `0-150`
- `gender`：必填，可选值：
  - `MALE`
  - `FEMALE`
  - `OTHER`
- `avatarId`：可选，仅存储，不参与当前 prompt 拼接

#### 响应体

```json
{
  "code": 200,
  "message": "OK",
  "data": {
    "userId": "usr_0d8df0d20b8c4bd8b6fca15f41c6d6ad",
    "sessionId": "sess_4f8f0d393a9a4d2788ef5525b5bbf603",
    "welcomeMessage": "你好张三，我是你的全科智能助手，请问有什么可以帮您？"
  }
}
```

#### 说明

- 若请求中不带 `userId`，后端生成 `usr_<32位uuid>`
- 不论是否带 `userId`，后端都会新建一个新的 `sessionId`
- `gender` 会在后端映射成内部画像值：
  - `MALE -> 男`
  - `FEMALE -> 女`
  - `OTHER -> 其他`

### 6.2 医疗聊天：同步 JSON

- 方法：`POST`
- 路径：`/api/v1/chat/completions`
- `Content-Type`：`application/json`
- `Accept`：`application/json`
- 响应：`application/json`

#### 请求体

```json
{
  "sessionId": "sess_4f8f0d393a9a4d2788ef5525b5bbf603",
  "message": "我这两天头晕",
  "attachments": []
}
```

#### 字段说明

- `sessionId`：必填，必须是已存在会话
- `message`：必填，用户本轮输入
- `attachments`：可选，但当前必须为空或不传

#### 响应体

```json
{
  "code": 200,
  "message": "OK",
  "data": {
    "sessionId": "sess_4f8f0d393a9a4d2788ef5525b5bbf603",
    "reply": "建议先休息，并注意补充水分。",
    "reportAvailable": true,
    "reportReason": "当前问诊信息较完整，可按需生成诊断报告。",
    "reportTriggerLevel": "suggested",
    "reportActionText": "当前问诊信息较完整，可按需生成诊断报告。",
    "reportGenerated": false,
    "report": null,
    "ragApplied": true,
    "sources": [
      {
        "sourceId": "kb-dizziness-triage",
        "title": "头晕分诊",
        "section": "危险信号",
        "score": 0.91
      }
    ]
  }
}
```

#### 返回字段说明

- `sessionId`：当前会话 ID
- `reply`：模型问诊回复
- `reportAvailable`：当前对话是否具备生成报告的条件
- `reportReason`：关于是否可生成报告的说明
- `reportTriggerLevel`：报告触发等级，取值为 `none | suggested | recommended | urgent`
- `reportActionText`：前端可直接展示的报告动作文案；当 `reportTriggerLevel=none` 时返回空字符串
- `reportGenerated`：本轮聊天是否已经直接带出报告
- `report`：若已生成报告，则返回 `ReportViewDto`，否则为 `null`
- `ragApplied`：本轮是否命中 RAG 检索增强
- `sources`：知识来源列表

#### 报告触发等级建议

- `urgent`：高风险或存在明确紧急就医信号，建议立即生成诊断报告协助就医
- `recommended`：已经形成较完整的医疗判断，建议生成诊断报告
- `suggested`：当前问诊信息较完整，可按需生成诊断报告
- `none`：当前仍处于追问/信息不足阶段，不建议前端主动提示生成报告

#### `sources` 字段说明

单项结构：

```json
{
  "sourceId": "kb-dizziness-triage",
  "title": "头晕分诊",
  "section": "危险信号",
  "score": 0.91
}
```

注意：

- 当前不会向前端暴露底层知识库文件路径
- 没有 `uri` 字段

### 6.3 医疗聊天：SSE 流式

- 方法：`POST`
- 路径：`/api/v1/chat/completions`
- `Content-Type`：`application/json`
- `Accept`：`text/event-stream`
- 响应：`text/event-stream;charset=UTF-8`

#### 请求体

与同步 JSON 版本完全相同：

```json
{
  "sessionId": "sess_4f8f0d393a9a4d2788ef5525b5bbf603",
  "message": "我这两天头晕"
}
```

#### 事件顺序

正常情况：

```text
event: meta
data: {"sessionId":"sess_xxx"}

event: chunk
data: {"delta":"建议先休息，","index":0}

event: chunk
data: {"delta":"并注意补充水分。","index":1}

event: done
data: {...完整 ChatCompletionResponse...}
```

异常情况：

```text
event: meta
data: {"sessionId":"sess_xxx"}

event: error
data: {"code":"SESSION_NOT_FOUND","message":"会话不存在"}
```

#### 事件说明

##### `meta`

```json
{
  "sessionId": "sess_xxx"
}
```

##### `chunk`

```json
{
  "delta": "建议先休息，",
  "index": 0
}
```

##### `done`

`done` 事件的 `data` 与同步 JSON 接口中的 `data` 完全同形：

```json
{
  "sessionId": "sess_xxx",
  "reply": "建议先休息，并注意补充水分。",
  "reportAvailable": true,
  "reportReason": "当前回复已经形成风险判断或排查方向，可生成诊断报告。",
  "reportGenerated": false,
  "report": null,
  "ragApplied": false,
  "sources": []
}
```

##### `error`

```json
{
  "code": "SESSION_NOT_FOUND",
  "message": "会话不存在"
}
```

#### SSE 前端接入注意事项

- 当前 SSE 是 `POST` 请求，不适合使用浏览器原生 `EventSource`
- 前端建议使用 `fetch` + `ReadableStream`
- 前端状态机应以 `meta -> chunk* -> done/error` 为准
- `done` 到达后，本轮问诊才算最终完成

### 6.4 查询问诊报告

- 方法：`GET`
- 路径：`/api/v1/reports/{sessionId}`
- 响应：`application/json`

#### 请求示例

```http
GET /api/v1/reports/sess_4f8f0d393a9a4d2788ef5525b5bbf603
```

#### 响应：报告尚不可生成

```json
{
  "code": 200,
  "message": "OK",
  "data": {
    "ready": false,
    "reason": "当前会话暂无足够问诊内容",
    "report": null
  }
}
```

#### 响应：报告可生成

```json
{
  "code": 200,
  "message": "OK",
  "data": {
    "ready": true,
    "reason": "",
    "report": {
      "title": "thread-1的医疗诊断报告",
      "riskLevel": "低风险",
      "summary": "头晕",
      "assessment": "考虑疲劳或轻度脱水",
      "basis": [
        "头晕"
      ],
      "recommendations": [
        "充分休息"
      ],
      "redFlags": [
        "意识模糊"
      ],
      "disclaimer": "本报告由AI生成，仅供参考，不能替代专业医生诊断。"
    }
  }
}
```

#### `ReportViewDto` 字段说明

- `title`：报告标题
- `riskLevel`：当前风险等级
- `summary`：患者概述
- `assessment`：初步判断
- `basis`：主要依据
- `recommendations`：建议下一步措施
- `redFlags`：需要警惕的升级信号
- `disclaimer`：固定免责声明

### 6.5 下载问诊报告 PDF

- 方法：`GET`
- 路径：`/api/v1/reports/{sessionId}/pdf`
- 响应：`application/pdf`

#### 请求示例

```http
GET /api/v1/reports/sess_4f8f0d393a9a4d2788ef5525b5bbf603/pdf
```

#### 成功响应

- HTTP 200
- `Content-Type: application/pdf`
- `Content-Disposition: attachment; filename="medical-report-{sessionId}.pdf"`
- `Cache-Control: no-store`

#### 失败响应

- `404 会话不存在`
- `409 当前会话暂无可导出的诊断报告`

## 7. 错误处理

### 7.1 JSON 接口错误格式

```json
{
  "code": 400,
  "message": "message 不能为空",
  "data": null
}
```

### 7.2 常见错误码与场景

#### HTTP 400

- `name 不能为空`
- `age 不能为空`
- `age 不能小于 0`
- `age 不能大于 150`
- `gender 不能为空`
- `sessionId 不能为空`
- `message 不能为空`
- `请求体格式不合法`
- `暂不支持附件上传`

#### HTTP 404

- `会话不存在`

#### HTTP 409

- `当前会话暂无可导出的诊断报告`

#### HTTP 500

- `问诊生成失败`
- `系统内部错误`

### 7.3 SSE 错误事件

SSE 不返回 `ApiResponse`，而是返回 `event:error`：

```json
{
  "code": "SESSION_NOT_FOUND",
  "message": "会话不存在"
}
```

当前已定义的 `error.code`：

- `SESSION_NOT_FOUND`
- `UNSUPPORTED_ATTACHMENTS`
- `CHAT_RUNTIME_ERROR`
- `INTERNAL_ERROR`

## 8. 当前前端联调建议

### 8.1 最小可用接入方式

推荐前端按照以下顺序联调：

1. `POST /api/v1/users/profile`
2. 保存 `userId + sessionId`
3. 用 `sessionId` 调用 `/api/v1/chat/completions`
4. 需要结构化结果时调用 `/api/v1/reports/{sessionId}`

### 8.2 前端本地需要保存的数据

- `userId`
- `sessionId`
- 当前聊天消息列表
- 当前报告状态

### 8.3 不要做的假设

前端当前不要假设以下能力已经存在：

- 不要假设支持附件上传
- 不要假设支持用户鉴权
- 不要假设 `sessionId` 可跨重启持久存在
- 不要假设 SSE 是模型原生 token 流
- 不要假设 `reportAvailable=true` 就一定同时返回 `report`

## 9. 当前实现边界

### 已支持

- 用户资料初始化
- 同步聊天
- SSE 聊天
- 报告查询
- RAG 来源回传

### 未支持

- 附件上传
- 登录鉴权
- 会话持久化
- 用户资料持久化
- 真正的 token 级模型流式输出

## 10. 给前端生成接口层时的建议

如果这份文档交给前端或 Claude 生成前端接口层，建议按以下方式抽象：

- `initProfile(payload): Promise<ApiResponse<ProfileInitResponse>>`
- `chatJson(payload): Promise<ApiResponse<ChatCompletionResponse>>`
- `chatStream(payload, handlers): Promise<void>`
- `getReport(sessionId): Promise<ApiResponse<ReportQueryResponse>>`

其中 `chatJson` 和 `chatStream` 使用同一个请求 DTO：

```ts
type ChatCompletionRequest = {
  sessionId: string;
  message: string;
  attachments?: string[];
};
```

SSE 事件类型建议前端固定为：

```ts
type ChatSseEvent =
  | { type: 'meta'; sessionId: string }
  | { type: 'chunk'; delta: string; index: number }
  | { type: 'done'; data: ChatCompletionResponse }
  | { type: 'error'; code: string; message: string };
```
