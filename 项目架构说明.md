# MedicalAgent 项目架构说明

## 1. 项目定位

`MedicalAgent` 是一个基于 `Spring Boot + Spring AI + Spring AI Alibaba Agent Framework` 的医疗问答后端原型项目，当前主要提供以下能力：

- 医疗问答对话
- 用户长期画像记忆
- 会话与线程管理
- 结构化医疗报告生成
- PDF 报告导出
- RAG 医疗知识库检索增强
- 就近医院推荐与路线规划

从代码组织上看，这个项目不是简单的 Controller + Service + Repository 三层结构，而是拆成了下面几条主链路：

- Web 接口链路：接收前端请求并输出稳定的 API 结构
- Agent 运行时链路：组织 Prompt、Hook、Interceptor 和模型调用
- 业务服务链路：聊天、报告、画像、会话等业务编排
- RAG 链路：知识入库、检索、上下文注入、离线评估
- 报告链路：报告生成、快照冻结、医院规划、PDF 导出

## 2. 核心分层总览

```text
前端请求
  -> controller / controller.v1
  -> session
  -> app.MedicalApp
  -> app.service.chat / profile / report / runtime
  -> hook / interceptor / app.rag.hook / app.rag.interceptor
  -> app.service.model + DashScope / MCP / VectorStore
  -> app.repository / app.repository.memory
  -> app.rag.*
  -> web.dto / web.support
```

可以把它理解为：

- `controller*` 负责“收口 HTTP”
- `MedicalApp` 负责“统一门面”
- `app.service.*` 负责“业务编排”
- `hook / interceptor` 负责“模型调用增强”
- `app.rag.*` 负责“知识检索增强”
- `app.report + app.service.report` 负责“结构化报告和导出”
- `repository / session` 负责“内存态状态管理”

## 3. 关键调用链

### 3.1 聊天链路

```text
ChatController
  -> ConsultationSessionService
  -> MedicalApp.doChat(...)
  -> DefaultMedicalChatService
  -> DefaultMedicalAgentRuntime
  -> MedicalRagAgentHook
  -> MedicalSystemPromptInterceptor / MedicalRagContextInterceptor / MyLogModelInterceptor
  -> DashScope ChatModel
  -> UserProfileMemoryHook
  -> ThreadConversationService
  -> MedicalApiViewMapper
```

这条链路的职责拆分是：

- `ChatController` 处理 JSON/SSE 两种聊天返回方式
- `ConsultationSessionService` 用前端 `sessionId` 找到内部 `threadId + userId`
- `MedicalApp` 统一收口业务入口
- `DefaultMedicalChatService` 判断是否是报告请求、执行聊天并封装结果
- `DefaultMedicalAgentRuntime` 真正组装 `ReactAgent`
- `MedicalRagAgentHook` 在模型前决定是否做知识检索
- `MedicalSystemPromptInterceptor` 和 `MedicalRagContextInterceptor` 负责拼接系统提示词
- `UserProfileMemoryHook` 在模型后回收用户长期信息
- `ThreadConversationService` 保存线程历史，供后续报告生成复用

### 3.2 报告链路

```text
ReportController
  -> MedicalApp.getOrCreateReportSnapshot(...)
  -> DefaultMedicalReportSnapshotService
  -> DefaultMedicalReportService
  -> DefaultMedicalHospitalPlanningService
  -> DefaultMedicalReportPdfExportService
  -> MedicalReportPdfRenderer
```

这条链路的核心思想是“先冻结快照，再复用快照”：

- 报告查询、报告预览、PDF 下载都尽量复用同一份 `MedicalReportSnapshot`
- 快照通过对话指纹、画像指纹、位置指纹判断是否仍然有效
- 如果快照失效，则重新生成报告并重新做医院规划
- 最终 PDF 导出只消费已经准备好的结构化报告数据

### 3.3 RAG 入库链路

```text
MedicalRagAdminController.reindex
  -> DefaultKnowledgeIngestionService
  -> MedicalKnowledgeDocumentLoader
  -> MetadataNormalizer
  -> TokenTextSplitter
  -> StableDocumentIdTransformer
  -> Summary / Keyword Enricher
  -> VectorStore.write(...)
  -> KnowledgeManifestRepository
```

这条链路负责把 `resources/knowledge-base/medical` 下的知识文档转成向量索引。

### 3.4 RAG 运行时链路

```text
MedicalRagAgentHook
  -> MedicalQueryBuilder
  -> RagTriggerPolicy
  -> RetrievalQueryEnhancer
  -> MedicalKnowledgeRetriever
  -> MedicalRagContextHolder
  -> MedicalRagContextInterceptor
```

它的特点是：

- 先判断这轮问题值不值得检索
- 再做检索查询增强
- 执行向量检索或 ES 混合检索
- 把命中的知识片段注入系统提示词
- 把来源列表回传给前端

## 6. 当前架构特点总结

从当前实现来看，这个项目有几个很鲜明的架构特征：

- 前端接口和内部 Agent 线程解耦：前端只操作 `sessionId`，后端内部维护 `threadId`
- `MedicalApp` 作为统一门面，避免 Controller 直接耦合多个服务
- RAG 不是简单依赖 Spring AI 默认 Advisor，而是深度嵌入到了 Agent Hook/Interceptor 流程
- 报告模块不是一次性生成文本，而是先输出结构化模型，再做预览、规划、导出
- 报告快照机制保证“查询报告、预览报告、下载 PDF”看到的是同一份内容
- 当前仓储大多是内存实现，说明项目仍偏原型/验证阶段，但接口抽象已经为后续换库留了位置

## 7. 官方基线与当前项目的自定义实现

### 7.1 官方参考基线

这个项目的 RAG 和向量库链路，和下面这些官方文档强相关：

- RAG 总体能力：<https://java2ai.com/integration/rag/retrieval-augmented-generation>
- ETL Pipeline：<https://java2ai.com/integration/rag/etl-pipeline/>
- Elasticsearch Vector Store：<https://java2ai.com/integration/rag/vectordbs/elasticsearch>

官方基线的核心思路是：

- 使用 `DocumentReader -> DocumentTransformer -> DocumentWriter(VectorStore)` 组织 ETL
- 使用 Spring AI 的 `VectorStore` / `ElasticsearchVectorStore` 作为向量库实现
- 在生成阶段通过标准 RAG 组件或 Advisor 注入检索上下文

### 7.2 当前项目保留的自定义点

当前项目没有完全照搬官方默认实现，而是保留了这些自定义设计：

1. `simple | elasticsearch` 双向量库切换  
   原因：本地开发希望支持轻量模式，联调或线上再切到 Elasticsearch。

2. `KnowledgeManifestRepository + StableDocumentIdTransformer`  
   原因：项目需要在重建知识库时先删除上一次写入的文档，避免脏数据残留。

3. `ElasticsearchBackedVectorStore` 包装层  
   原因：项目希望手动控制 ES store 的初始化与启动失败回退逻辑，避免容器启动阶段直接因 ES 不可达而失败。

4. `ElasticsearchHybridSearchClient` 应用层混合检索  
   原因：项目需要在标准向量检索之上叠加 BM25 词法检索和 RRF 融合，提高医疗场景下的召回稳定性。

5. `MedicalRagAgentHook + MedicalRagContextInterceptor`  
   原因：项目基于 Agent Framework 的 `ReactAgent` 运行，需要把 RAG 过程嵌进 Agent 生命周期，而不是只停留在标准 ChatClient Advisor 层。

## 8. 一句话理解整个项目

如果用一句话概括当前架构：

> 这是一个以 `MedicalApp` 为统一门面、以 `ReactAgent` 为执行内核、以 `RAG + 结构化报告 + 医院规划` 为特色能力的医疗问答后端项目。
